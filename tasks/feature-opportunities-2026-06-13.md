# DiaBloS Modern — Improvement & New-Feature Opportunities

**Date:** 2026-06-13  
**Method:** 8 parallel exploration agents surveyed the codebase (blocks, engine/solvers, control & analysis, data/interop, UX, PDE/scientific, performance, platform), each grounding proposals in actual files and flagging already-roadmapped items; one synthesis pass prioritized the results.  
**Raw opportunities:** 86 → deduped into the roadmap below.

## Executive summary

DiaBloS Modern is a mature PyQt5 block-diagram simulator with a compiled scipy solve_ivp fast path, a rich block library, live parameter tuning, and TikZ/animation export. The ~80 opportunities collapse into roughly a dozen distinct bets once deduped: Data Import/FromFile appears 3x; numerical linearization appears 4x; parameter sweep + Monte Carlo + sensitivity are one re-sim-engine family; code generation appears 2x; broaden compiled-block coverage and sparse Jacobian each appear 2x. The most important structural finding, confirmed in the code: lib/analysis/linearizer.py has a fully-implemented analysis suite (transfer function via ss2tf, eigenvalue/stability analysis, controllability/observability matrices) that is unreachable because linearize_at_point() is a deliberate fail-loud NotImplementedError stub. Meanwhile system_compiler.compile_system() already returns the (model_func, y0, state_map, block_matrices) artifact needed to finite-difference an A/B/C/D. Numerical linearization is therefore the keystone: cheapest path to activate the largest cluster of dormant value (Bode/Nyquist/root-locus/pole-zero/margins on ANY nonlinear plant), and already on the roadmap. The roadmap prioritizes keystone unlocks, quick wins riding existing plumbing, medium-effort high-impact features for the teaching/research/control+PDE audience, and strategic platform bets. Teaching value is weighted heavily; large engine rewrites without near-term teaching payoff are discounted.

## Top recommendation

Build the numerical linearization tool wired to the compiled ODE RHS (lib/analysis/linearizer.py linearize_at_point on top of system_compiler.compile_system's model_func/state_map). I verified this is the highest leverage-to-effort move available: linearize_at_point() is a deliberate fail-loud stub, yet the entire downstream analysis suite it would feed — compute_transfer_function (scipy ss2tf), analyze_stability (eigenvalues, damping, time constants), and controllability/observability matrices — is already fully implemented and tested in the same file, sitting dormant and unreachable. The compiler already hands back the exact (model_func, y0, state_map) artifact needed to finite-difference an A matrix (compute_jacobian_numerical even exists), and get/set_state_vector already round-trip per-block state. So a comparatively small amount of wiring (perturb y for A, perturb input sources for B, read outputs for C/D at an operating point) unlocks Bode, Nyquist, root-locus, pole-zero maps, and stability margins for ANY nonlinear diagram — tank, pendulum, multi-agent — instead of only already-linear blocks. It is already on the roadmap (todo.md 'Linearization tool'), directly serves the core teaching/research/control audience and the c01..c11 course examples, and is the foundation the trim finder, the unified analysis dashboard, PID auto-tuning, the MAT/LaTeX matrix exporters, and empirical-vs-analytical frequency-response comparison all build on. Pair it with the trim/operating-point finder (medium effort) for a turnkey find-equilibrium-then-linearize-then-analyze workflow. The genuine quick wins (finish the Export block, chirp source, Bode margins + auto-range, solver diagnostics, Demux/Logic compiled executors) should run in parallel as they are nearly free and independently shippable.

## Quick wins (high value, low effort)

- **Finish the Export block: route to CSV/MAT/NPZ via a format param** — Doc advertises .npz/.mat/.csv but export_data() (verified line 1220) only calls np.savez — a documented, broken promise. scipy.io.savemat and np.savetxt are trivial over the vec_dict/labels structure already assembled; pandas/pyarrow already bundled in diablos.spec.  
  _Leverages:_ lib/lib.py:export_data() vec_dict assembly; scipy already a dep; waveform_inspector._export_csv np.savetxt pattern.
- **Chirp / swept-sine source block** — Standard excitation for empirical frequency-response labs; into the existing FFT and Bode blocks it produces a measured Bode plot end-to-end, which fixed-frequency sources cannot. Net-new but tiny.  
  _Leverages:_ wave_generator.py already imports scipy.signal (verified line 4); chirp is in the same module; source b_type=0 pattern; downstream fft.py/bodemag.py.
- **Gain/phase margins + auto-ranging frequency axis on Bode/Nyquist** — Margins are the core quantitative takeaway of a Bode/Nyquist plot — without them the plots are decorative. Hard-coded logspace(-2,2) range leaves fast/slow poles off-screen. Both ride frequency-response data already computed.  
  _Leverages:_ bode.py/nyquist.py (mag/phase already computed; reference lines drawn); np.roots pole/zero magnitudes already used in root_locus.py.
- **Solver diagnostics panel (nfev, steps, rejected steps, wall time)** — Turns the solver into a teaching instrument — students SEE why a stiff system needs 10x more RHS evals. scipy returns nfev/status/message on the sol object but it is thrown away today (only a string reaches the UI).  
  _Leverages:_ run_compiled_simulation already holds the scipy sol object (verified line 1224); simulation_controller already emits status_changed.
- **Broaden compiled-path coverage: Demux + Logic blocks** — check_compilability falls back to the slow interpreter if ANY block is unsupported, so one Demux or logic gate forfeits the 10-100x speedup for the whole diagram. Demux is symmetric with the compiled Mux (line 69 notes 'No compiled executor yet'); Logic blocks (added 2026-06-02) are pure functions of inputs.  
  _Leverages:_ exec_mux as the symmetric template; COMPILABLE_BLOCKS allowlist (verified lines 52-120) + elif dispatch is the single extension point.
- **Copy/paste parameters across same-type blocks** — Course diagrams have repeated identical controllers/sensors; today the property editor only resets-to-default and the clipboard copies whole blocks. Two-click 'tune one, replicate' using the deepcopy clipboard_manager already performs on block.params.  
  _Leverages:_ property_editor.py _reset_param/_categorize_params; menu_manager block context menu; clipboard_manager deepcopy of params already done.
- **Goto/From tag navigation + unify the .diablos extension** — Two small fixes. A Goto/From HUD already enumerates tags but has no click-through to the partner block (key for consensus examples). FileService defaults to data.dat and dialogs filter *.dat while every example uses .diablos, so students cannot see their own files in the open dialog.  
  _Leverages:_ canvas_renderer.draw_tag_hud (tags enumerated); existing error-jump pan/center code; file_service dialog filters; menu_builder already lists .diablos.
- **Refresh stale contributor docs + auto-generate in-app block reference in CI** — DEVELOPER_GUIDE teaches a contract that no longer exists (references deleted lib/functions.py and @property attributes), contradicting the authoritative CLAUDE.md contract. tools/generate_docs.py already walks the registry and emits Markdown — promote it to a CI drift-check and in-app panel. Biggest contributor onboarding trap, near-zero effort.  
  _Leverages:_ tools/generate_docs.py registry walk; blocks/base_block.py doc/params; .github/workflows/ci.yml.

Pairs with a gain-from-port block for gain scheduling. scipy.interpolate already a dep.
- **Resettable Integrator/StateSpace with external-IC and reset port** _(effort: medium — integrator.py multi-method path; PID anti-windup clamp as reference; state_map isolates per-block y slots for the compiled reset event.)_ — Reset integrators are needed for hybrid/switching controllers, PWM, sample-and-hold, and bumpless transfer. Dedup of the blocks-dimension and engine-dimension reset entries.
- **Sparse/banded analytic Jacobian for stiff PDE solving (+ vectorize 2D BC loops)** _(effort: medium — state_map + stencils have known band structure; scipy already a dep; edge-loop vectorization mirrors landed 1D work.)_ — PDE systems are hundreds-to-thousands of states with known band structure, but solve_ivp gets no Jacobian (verified: no jac= at line 1224), so Radau/BDF estimate a dense FD Jacobian — O(n^2) RHS evals/step. jac_sparsity from state_map+stencils can speed implicit runs 10-100x — the payoff of the shipped solver-selection feature. Edge/BC loops (on the backlog) are the compounding small win.
- **Compiled-system caching across re-runs** _(effort: medium — compile_system + run_compiled_simulation + tuning_controller debounce loop are the localized touch points.)_ — run_compiled_simulation calls compile_system() every run, rebuilding closures, re-parsing matrices, and re-running the Kahn sort even when one scalar changed. Caching keyed on topology so only changed scalars re-bake makes live tuning (50ms loop) and the proposed sweep dramatically more responsive.

## Strategic bets (large)

- **Zero-crossing / event detection in the compiled solver** — Wiring solve_ivp events= (verified absent) lets discontinuous blocks land a sample exactly on each crossing instead of RK45 smearing/skipping it. The principled fix that re-admits Impulse/Step-impulse/Hysteresis to the fast path (currently excluded, verified lines 59-92) and the substrate for resettable integrators. Transformative for relay/bang-bang accuracy, but a substantial engine change.
- **Export diagram as standalone runnable Python script (code generation)** — compile_system() already lowers a diagram to a flat RHS closure + y0 + metadata, so codegen is largely serializing that to source plus a solver/plot stub. Lets users share runnable models with collaborators lacking DiaBloS and embed reproducible models in papers. Dedup (appears 2x). Already on the roadmap. Large because every stateful block must emit correct source.
- **Third-party block plugin SDK + safe sandboxed External block** — Turns DiaBloS from a closed app into an extensible platform: pip-installable block packs (QCar2 interface, lab PDE kernels) appearing in the palette without forking. Discovery is hard-coded to in-repo blocks + static _BLOCK_MODULES. Pairs with re-implementing the disabled External block via the proven safe_expr sandbox (external.py spells out the security requirement). High ceiling, pays off once an external author exists.
- **New PDE physics blocks: Burgers, 2D Gray-Scott, Schrodinger, Maxwell 1D (+ absorbing/PML BCs)** — A family of net-new blocks expanding the surface beyond decaying-heat-blob demos: Burgers (canonical nonlinear-conservation bridge), Gray-Scott Turing patterns (visually striking 2D), Schrodinger tunneling (physics), Maxwell/FDTD (EE). Each reuses existing stencil/BC/state machinery. Sequence AFTER the small CFL advisor and Crank-Nicolson work so new blocks don't silently blow up. Absorbing/PML and Periodic-BC-everywhere already on the roadmap.
- **Multiple open diagrams via tabs + canvas viewport culling** — Tabs unlock side-by-side coursework (open-loop vs LQR) and cross-diagram reuse, replacing today's single-document model. Viewport culling (paintEvent already computes scene_rect but render loops repaint everything) makes large diagrams pan/zoom smoothly. Both sizable canvas/main-window refactors; valuable but not blocking the core teaching loop.
- **Semi-explicit index-1 DAE / algebraic-loop solve** — Today an algebraic loop is a hard error. Solving the implicit relation per step with a small Newton iteration (Newton code exists in optimization_primitives) would enable constrained/force-balance models and direct-feedthrough feedback that several shipped optimization blocks imply. Transformative but niche for the teaching audience — sequence last.

## Full opportunity catalog (by dimension)

### Block library

- **1-D and 2-D Lookup Table block (interpolation)** — _new-feature, effort: medium_  
  A LookupTable block that maps an input through a user-supplied breakpoint/value table using linear (and optionally nearest/cubic) interpolation, with selectable extrapolation (clip vs. linear). Support 1-D (one input) and 2-D (two inputs) variants. scipy.interpolate is already a dependency.  
  **Value:** Lookup tables are the standard Simulink way to model nonlinear actuator/sensor curves, gain-scheduled controllers, valve characteristics, and empirically measured plant maps. This is the single most common missing block for control teaching and for matching real hardware like QCar2.  
  **Leverages:** blocks/base_block.py contract; scipy already imported (blocks/integrator.py, blocks/wave_generator.py); param patterns in blocks/param_templates.py; selector.py already parses list/range strings for index parsing; safe_literal in lib/safe_eval.py for parsing table arrays as used in system_compiler.py
- **Data Import / From-File source block (CSV/MAT time series)** — _new-feature, effort: medium_ _(already on roadmap)_  
  A source block that reads a time-series from a CSV/NPZ/MAT file and replays it as a signal, interpolating to the simulation time grid (zero-order-hold or linear). Columns selectable; loops or holds at end.  
  **Value:** Essential for validating simulations against experimental/recorded data (e.g. QCar2 logs) and for model fitting. Pairs directly with the existing optimization blocks (data_fit/cost_function) to do data-driven parameter identification, which currently has no way to inject real measurements.  
  **Leverages:** lib/export already does CSV/npz I/O (lib/export/); blocks/optimization/data_fit.py expects a reference signal; transport_delay.py already implements binary-search time interpolation that can be reused; b_type=0 source pattern from constant.py/ramp.py
- **Integrator with reset, external initial condition, and exposed anti-windup limits** — _improvement, effort: medium_  
  Extend the existing Integrator with optional input ports: a reset trigger (rising-edge / level) that re-seeds state to the IC, and an external-IC port. Also surface the saturation limits the docstring already promises (the doc mentions 'Limit Output' but params expose no min/max, so output limiting is undocumented-missing).  
  **Value:** Reset integrators are needed for hybrid/switching controllers, PWM, sample-and-hold logic, and bumpless controller switching taught in control courses. The doc/param mismatch is also a latent correctness gap: students reading the help expect output limiting that does not exist.  
  **Leverages:** blocks/integrator.py (existing multi-method integrator and SOLVE_IVP path); limit_params() in blocks/param_templates.py; b_type=1 memory-block handling; reset_memblocks pattern noted in CLAUDE.md; PID anti-windup clamp logic in blocks/pid.py as a reference implementation
- **Kalman Filter / Luenberger Observer block** — _new-feature, effort: large_  
  A block parameterized by (A,B,C,D) plus observer gain L (Luenberger) or process/measurement covariances Q,R (steady-state or time-varying Kalman). Takes (u, y) inputs and outputs the state estimate x_hat. Could compute L/K via existing scipy linalg.  
  **Value:** The c06 observer example is currently hand-built from two StateSpace blocks plus a manual gain feedback loop, which is fragile and hard for students to assemble. A first-class observer block makes estimation labs (observer design, output feedback, sensor fusion) one drag-and-drop and complements the existing LQR block for full LQG teaching.  
  **Leverages:** blocks/statespace_base.py (matrix validation, state init, x[k+1]=Ax+Bu update); blocks/lqr.py (existing Riccati/scipy.linalg usage pattern); examples/c06_observer_estimation.diablos as the reference design to subsume
- **Unit Delay / Memory block (z^-1) with sample time** — _new-feature, effort: small_  
  A dedicated single-step Memory/UnitDelay block that outputs the previous sample (z^-1) and is the canonical algebraic-loop breaker. Distinct from the existing N-step Delay because it carries explicit sample-time semantics and an initial-condition port, and is the building block for discrete controllers.  
  **Value:** z^-1 is the fundamental discrete-time primitive students use to build difference equations, IIR/FIR filters, and discrete observers from scratch. The current N-step Delay works but is not recognized as the idiomatic loop-breaker / discrete memory, and lacks sample-time semantics for multi-rate diagrams.  
  **Leverages:** blocks/delay.py (circular-buffer pattern, b_type=1 loop-breaking already implemented); sampling_time param convention from blocks/integrator.py and blocks/pid.py; rate_transition.py multi-rate handling
- **Quantizer / Saturating-quantizer block (ADC/DAC modeling)** — _new-feature, effort: small_  
  A Quantizer block with configurable step size and rounding mode (round/floor/ceil/truncate), optionally combined with saturation, modeling finite-resolution sensors and DACs/ADCs.  
  **Value:** Quantization is a core topic in digital control and signal processing courses (effect of ADC resolution on stability, limit cycles). It pairs naturally with the existing ZeroOrderHold/RateTransition blocks to demonstrate realistic sampled-data control of the QCar2 and other digital plants.  
  **Leverages:** blocks/saturation.py (clip_to_limits in blocks/input_helpers.py); element-wise numpy pattern shared across math blocks; MathFunction already has floor/ceil/sign primitives to mirror
- **Chirp / swept-sine source block** — _new-feature, effort: small_  
  A frequency-sweep source (linear/logarithmic chirp) with start/stop frequency over a sweep time, plus optional amplitude envelope. scipy.signal.chirp is already available.  
  **Value:** Swept sines are the standard excitation for empirical frequency-response / system-identification labs. Feeding a chirp into a plant and into the existing FFT and BodeMag/BodePhase blocks gives students a measured Bode plot end-to-end, which the current fixed-frequency sources (sine, wave_generator) cannot produce.  
  **Leverages:** blocks/wave_generator.py already imports scipy.signal and uses signal.square/sawtooth (chirp lives in the same module); blocks/fft.py and blocks/bodemag.py for the downstream frequency-response workflow; source b_type=0 pattern
- **MinMax / running min-max / argmax block** — _new-feature, effort: small_  
  A configurable block computing element-wise or across-input min/max (and optionally running min/max over time, plus index/argmax output). Variable input-port count.  
  **Value:** min/max selection is needed for override/selector control, constraint switching (e.g. min of two controller demands), saturation-by-comparison, and consensus/multi-agent demos (c09/c10). Today users must hand-build this with relational + switch blocks, which is verbose and error-prone for teaching.  
  **Leverages:** blocks/switch.py io-editable variable-port pattern (io_editable, get_inputs); blocks/relational_operator.py element-wise comparison logic; blocks/logical_operator.py variable-input handling
- **Variable / input-driven transport delay (port-controlled delay time)** — _improvement, effort: small_  
  Add an optional second input port to TransportDelay that sets the delay time dynamically at run time (variable dead time), instead of only a fixed delay_time parameter.  
  **Value:** Variable transport delay models flow-rate-dependent pipe/conveyor lag and speed-dependent vehicle dead time (relevant to QCar2). It is a standard Simulink 'Variable Transport Delay' block and unlocks teaching the destabilizing effect of changing dead time on feedback loops.  
  **Leverages:** blocks/transport_delay.py already has the full interpolating ring-buffer history; only needs an optional input port and to read target_time from inputs[1]; optional_inputs set support in blocks/base_block.py
- **Time-varying / scheduled gain block (gain from input port)** — _improvement, effort: small_  
  Add a Gain mode that takes the gain (scalar or matrix) from a second input port rather than a fixed parameter, enabling external/scheduled multiplication. Effectively a clean element-wise/matrix multiply where one operand is a gain signal.  
  **Value:** Gain scheduling and adaptive control require a gain that changes during simulation. Today the only path is the generic Product block, which is non-obvious for matrix-times-vector scheduling and loses the visual semantics of a gain. This is the natural companion to the proposed LookupTable for building gain-scheduled controllers.  
  **Leverages:** blocks/gain.py and blocks/matrix_gain.py (existing scalar/matrix multiply); blocks/product.py and blocks/sigproduct.py for the two-operand pattern; optional_inputs support in base_block.py
- **Finite State Machine / discrete state-machine block (Stateflow-like)** — _new-feature, effort: large_  
  A block defining discrete states with guard conditions on inputs (using the existing safe_expr sandbox) and per-state outputs/actions; emits the active state index and state-dependent outputs. Start with a transition-table form before any graphical editor.  
  **Value:** Supervisory/mode logic (start-up sequences, fault handling, hybrid switching controllers, traffic-light and bang-bang demos) is currently impossible without chaining many relational/switch/logic blocks. A FSM block is a major teaching capability for hybrid and event-driven control that the library completely lacks today.  
  **Leverages:** lib/safe_eval.py safe_expr AST sandbox already used by blocks/function.py and math_function.py for guard evaluation; blocks/function.py io_editable variable-port pattern; logical_operator.py/relational_operator.py for condition primitives; b_type=1 memory semantics from delay.py for state persistence
- **Ramp-to-value / signal-builder source (piecewise reference)** — _improvement, effort: medium_  
  Extend or add to Ramp a 'ramp to final value then hold' mode, and/or a piecewise SignalBuilder source defined by (time, value) breakpoints with linear interpolation. The current Ramp grows unbounded with no final value.  
  **Value:** Trapezoidal/piecewise reference trajectories (rate-limited setpoint changes, motion profiles) are the realistic test signals for tracking labs and avoid the unbounded-ramp pitfall. A SignalBuilder also lets instructors craft custom test scenarios without scripting.  
  **Leverages:** blocks/ramp.py (existing slope/delay ramp); transport_delay.py time-interpolation helper for the breakpoint interpolation; rate_limiter.py for the rate-limited-setpoint variant; the proposed LookupTable shares the breakpoint-parsing code

### Solvers & engine

- **Zero-crossing / event detection in the compiled solver** — _new-feature, effort: large_  
  Wire scipy.integrate.solve_ivp's events= argument into run_compiled_simulation so discontinuous blocks register event functions (Saturation hitting a limit, Switch crossing its threshold, Deadband edges, Step/Impulse delay time, Hysteresis relay switch). The solver lands a sample exactly on each crossing and restarts integration, instead of the adaptive RK45 stepping over the discontinuity. This is also the principled fix that lets Impulse/Step-impulse and Hysteresis re-enter the compiled path.  
  **Value:** Control students see crisp, correctly-timed responses for relay/bang-bang controllers, saturating actuators, and switched systems instead of smeared or skipped transitions. Today the code explicitly excludes Impulse, Step(impulse), Hysteresis and Switch-heavy behavior from the fast path precisely because adaptive stepping corrupts them (system_compiler.py lines 59-118).  
  **Leverages:** lib/engine/simulation_engine.py run_compiled_simulation (solve_ivp call ~line 1224 currently has no events=); per-block executor closures in lib/engine/system_compiler.py (Saturation ~539, Switch ~589, Deadband ~548, Hysteresis ~884, Step/Impulse ~352-383) already know their thresholds and could emit g(t,y) event functions.
- **Stiff-solver auto-selection with fallback on RK45 failure** — _improvement, effort: medium_  
  Add an 'Auto' solver mode that inspects the compiled system for stiffness indicators (large Integrator/StateSpace count, PDE blocks, the 1000.0 penalty-method Dirichlet/Robin BCs that make heat/diffusion systems numerically stiff) and starts with Radau/BDF instead of RK45; additionally, if an adaptive run returns sol.success==False, automatically retry once with LSODA before reporting failure.  
  **Value:** PDE demos (heat/diffusion with penalty BCs) and stiff control plants currently force students to know to manually pick Radau/BDF from the dropdown or watch RK45 crawl/fail. Auto-selection makes the default 'Run' button just work for the teaching diagrams shipped in examples/.  
  **Leverages:** Stiff solvers Radau/BDF/LSODA are already selectable (simulation_engine.py SCIPY_SOLVER_METHODS line 21, dialog SOLVER_METHODS in lib/dialogs.py line 76); penalty-method BCs in system_compiler.py (e.g. heat exec_heat1d line 957, 1000.0 factor) are a known stiffness source; sol.success is already checked at line 1227.
- **Solver diagnostics panel (steps, RHS evals, rejected steps, wall time)** — _new-feature, effort: small_
